About

4 min read
sitenextjsmarkdownstatic-site

A Next.js static site template for intelligent, agent-friendly publishing from markdown.

Point it at a directory of markdown files, run a single command, and get a fully built static website ready for GitHub Pages.

Purpose

nlp-mdsite provides a minimal, mobile-first site template designed to work at the end of a markdown publishing pipeline. The goal is a push-button workflow: point the pipeline at a directory of markdown files, run a single command, and get a fully built, deployed website.

The template is designed to be configured as easily by an AI agent as by a human — a small config change is all that's required to launch a new site.

Get Started

See Getting Started to have a site running in minutes, or browse the Features section for the full capability overview. For background on the project and its roadmap, see About.

Features

  • Markdown → MDX — automatic conversion, any folder structure
  • Images — copied and path-rewritten automatically; corrupt EXIF data stripped
  • Reading time — estimated and injected into every page's frontmatter
  • Tags and categories — rendered as pill chips below each title and in the sidebar
  • Sidebar metrics — any numeric frontmatter field surfaces as a labeled score
  • Nav ordering — configure page and folder order via site.config.js or frontmatter order:
  • Per-page feed — scroll to the bottom of any page to load the next one inline
  • Theme toggle — light / dark / system toggle in the navbar
  • GitHub header icon — circular GitHub repo link, auto-shown from repo_url in config
  • GitHub Pages — one workflow, zero manual steps

Intelligence Layer

The "NLP" in nlp-mdsite refers to a set of built-in and planned intelligent features:

  • Rich metadata display — tags, reading metrics, and other signals generated by the upstream pipeline are displayed as first-class UI elements on each page
  • Semantic search (planned) — search across all content by meaning, not just keywords
  • Semantic theming (planned) — the site's visual identity is automatically derived from content signals rather than manually specified

These features are intended as defaults, not plugins. The upstream pipeline (mdpub) is responsible for generating the metadata and semantic signals; this template consumes and renders them.

How It Works

  1. Write content as markdown files in any folder structure
  2. Run npm run ingest to mirror content into pages/, rewrite paths, and generate indexes
  3. Run npm run build to produce the static site in out/
  4. Deploy to GitHub Pages (or any static host) — included workflow does this automatically

Getting Started

Prerequisites

  • Node.js (opens in a new tab) 18 or later
  • npm (bundled with Node.js)

Setup

1. Install dependencies

npm install

2. Add your content

Place your markdown files in a source directory in any folder structure. The directory tree is mirrored directly into the site — subdirectories become URL path segments.

<source>/
├── home.md          # home.md or index.md at any level → section landing page
├── about-us.md
├── images/          # images/ next to any .md files → copied to public/images/
└── posts/           # any subdirectory structure is valid
    └── <year>/
        ├── post-slug.md
        └── images/

Frontmatter fields used by the template: title, date, order, categories, tags.

3. Configure the site

Edit site.config.js to set the site title and base URL:

module.exports = {
  title: 'My Site',
  base_url: 'https://username.github.io',
  base_path: '/repo-name',   // empty string for root domains
  repo_url: 'https://github.com/username/repo-name',
}

4. Ingest content

npm run ingest docs            # default: ingest from docs/
npm run ingest path/to/source  # ingest from a custom directory

This mirrors markdown into pages/, renames .md → .mdx, copies images to public/images/, rewrites image and link paths, injects reading time, and generates _meta.json ordering files, posts-index.json, and feed-index.json. It also syncs README.md to the /about page.

5. Preview locally

npm run dev        # development server with hot reload
npm run build      # production build → out/
npm run preview    # serve the built out/ directory

Publishing to GitHub Pages

The included workflow (.github/workflows/deploy.yml) builds and deploys automatically on every push to main. Follow these steps to enable it for the first time:

1. Push to GitHub

git add .
git commit -m "Add docs content and configure deployment"
git push

2. Enable GitHub Pages (one-time setup)

Go to your repository on GitHub:

  • Settings → Pages → Build and deployment
  • Set Source to Deploy from a branch
  • Set Branch to gh-pages, folder / (root)
  • Click Save

The gh-pages branch is created automatically by the workflow on first run.

3. Set the BASE_PATH variable (project pages repos only)

If your site lives at a subpath (e.g. username.github.io/repo-name), set this variable:

  • Settings → Secrets and variables → Actions → Variables → New repository variable
  • Name: BASE_PATH, Value: /repo-name (e.g. /nlp-mdsite)

Skip this step if you are deploying to a root domain (username.github.io).

4. Watch the workflow

Go to the Actions tab on GitHub. The "Deploy to GitHub Pages" workflow should be running (or complete). A green checkmark means the deploy succeeded.

5. Access the live site

Your site will be live at:

https://<username>.github.io/<repo-name>/

GitHub Pages can take 1–2 minutes to propagate after the first deploy.

Integrations

nlp-mdsite is the rendering and build step at the end of the mdpub (opens in a new tab) publishing pipeline. Configuration, API interfaces, and MCP tooling live in mdpub; this template is driven entirely by its config file and content directory.

module.exports = {

source/

npm run setup -- --source path/to/source

This ingests content, rewrites image paths, and generates navigation files.

npm run dev

Frontmatter fields

Add these to the top of any markdown file to control how it is displayed:

---

  - tutorial

Nav ordering

By default the pipeline sorts pages newest-first by date, or alphabetically. Use nav_order to define explicit ordering at any directory level:

nav_order: {
  // '' = source root; 'features' = the features/ subdirectory
  '': ['getting-started', 'configuration', 'features', 'posts'],
  'features': ['content-pipeline', 'metadata', 'deployment'],
},

Folders and pages can be mixed in the same list. Slugs not listed append alphabetically after the explicit entries.

For individual pages without a full directory listing, add order: N to the page's frontmatter instead — lower numbers appear first, pages without order sort after.

GitHub Actions variables

The deployment workflow reads two optional repository variables. Set them under Settings → Secrets and variables → Actions → Variables.

VariableDefaultDescription
CONTENT_SOURCEdocsPath to content directory, relative to repo root
BASE_PATH(empty)GitHub Pages base path, passed to Next.js at build time

BASE_PATH is required whenever your site lives at a subpath (e.g. username.github.io/repo-name). Leave it empty for root domain deployments.

Components

2 min read
developmentcomponentsreactcustomization

All UI components live in components/. They are wired into the site via theme.config.jsx and styled in styles/global.css. No imports are needed inside markdown — components that appear on every page are registered globally in theme.config.jsx.

ComponentFileWhere rendered

Component reference

PageHeader

Renders publication date and estimated reading time on a single line below the page title. Returns null when neither date nor reading_time is present in frontmatter.

Props: date (YYYY-MM-DD string), reading_time (integer minutes).

TagList

Renders categories as blue chips and tags as gray chips below PageHeader. Returns null when both arrays are empty.

Props: categories (string[]), tags (string[]).

MetaSidebar

Sticky right-hand sidebar with three sections, each rendered only when data is present:

  • Tags — categories (blue) and tags (gray) from frontmatter
  • Metrics — any numeric frontmatter field not in the reserved set (title, date, categories, tags, reading_time, related)
  • Related — related frontmatter array of { title, url } objects

Hidden below 1024 px via CSS media query. Reads frontmatter via useConfig().

SiteFooter

Renders the site-wide footer: copyright year, build timestamp, and credits. Edit components/SiteFooter.jsx directly to customize footer content across all pages.

GitHubLink

Circular GitHub mark icon in the navbar. Returns null when repo_url is empty in site.config.js. No props — reads config directly.

ThemeToggle

Cycles through system → light → dark using useTheme() from next-themes. Shown in the navbar when theme_toggle: 'navbar' in site.config.js; otherwise Nextra's built-in sidebar toggle is used. Uses a mounted-state guard to prevent hydration mismatch.

PageContinuation

Per-page continuation feed. When the user scrolls to the bottom of a page, the next page's content is loaded inline using IntersectionObserver. Requires:

  • feed: true in site.config.js
  • public/feed-index.json (generated by ingest.js)
  • react-markdown dependency (included)

Enabled/disabled globally via siteConfig.feed in theme.config.jsx.

PostIndex

Fetches public/posts-index.json at runtime and renders a listing of all dated posts with title, date, reading time, and category chips. Used on the auto-generated posts/index.mdx page when dated posts are found during ingest.

Adding a new component

1. Create the component file

// components/MyWidget.jsx
/** One-line description of what this component does. */
export default function MyWidget({ label }) {
  return <div className="my-widget">{label}</div>
}

2. Add CSS

Add styles to styles/global.css. Use --nextra-colors-* variables for theme-adaptive colors. See Styling for details.

.my-widget {
  padding: 0.5rem 1rem;
  color: var(--nextra-colors-gray-600, #4b5563);
}

3. Wire into theme.config.jsx

For components that appear on every page, import them in theme.config.jsx and add them to the main layout or navbar.extraContent:

 
 
// In the main layout:
main: ({ children }) => (
  <div className="page-layout">
    <div className="page-content">
      <PageMeta />
      {children}
      <MyWidget label="hello" />
    </div>
    <MetaSidebar />
  </div>
),

For components used only on specific pages, import them directly inside the .mdx file:

 
 
# My Page

4. Access frontmatter (optional)

Use useConfig() from nextra-theme-docs inside the component to read the current page's frontmatter:

 
 
export default function MyWidget() {
  const { frontMatter } = useConfig()
  return <div>{frontMatter.title}</div>
}
